That's just the way I do Binary Search. There are slightly other approaches, but I like mine and it works for me.

Let's discuss it with a simpler example. Given a sorted array aa I want to find the number xx. I actually would convert it into a slightly more complicated problem. Find the first index ii with a[i]≥xa[i]≥x.

This splits the array into two parts. The first few elements are smaller than xx, so the condition a[i]≥xa[i]≥x is not satisfied. We can call these the bad elements. The other elements all satisfy the condition. We can call them the good elements.

This simplifies the problem for me. Instead of having to think about lots of different numbers, I just have good and bad ones, and I'm looking for the index, where the bad numbers turn into good ones. In other words I'm looking for the smallest good number.

Now I can think about LL and RR. It's very important in binary search to be very careful and clearly define variables. Otherwise you can very easily miss some corner case.

For that I split the array into 3 parts. The numbers [0;L][0;L] are the bad ones. The numbers [R,n−1][R,n−1] are the bad ones. And the other ones [L+1,R−1][L+1,R−1] I don't know yet.

I always keep the two invariants: all indices ≤L≤L are bad, and all indices ≥R≥R are good. Now what about the initialization. I don't know if a[0]≥xa[0]≥x. So I can't initialize LL with 00. Instead, I have to initialize it with 00. The same with RR. Since I don't know yet if a[n−1]≥xa[n−1]≥x, I assign it with nn.

This gives me the code:

int L = -1; // bad numbers, < x
int R = n;  // good numbers, >= x
while (L + 1 < R) {
    int M = (L + R) / 2;
    if (a[M] >= x)
        R = M;
    else
        L = M;
}
cout << R << '\n';

I do the iteration as long L+1<RL+1<R (because L+1=RL+1=R would mean I have already classified every index as good or bad). This means R−L≥2R−L≥2, and therefore MM will be a number that is strictly greater than LL and strictly smaller than RR, so a number that I haven't classified as good or bad.

At the end RR is the smallest good index, so RR is the answer.

Maybe that sounds a little strange to you, but this is how I think and it works great for me. When I started with competitive programming, I sometimes struggled with binary search. So back then I figured out that way, and I never had any problems again.

But that doesn't mean that you also have to do it like that. A lot of people do it slightly differently, and it also works for them.

I actually always, even in easy problems, clearly declare how and what LL and RR means. E.g. in this solution to a Div2 problem 1011B, I make LL as possible (this are the good numbers, the potential results), and RR as not possible (the bad numbers). In that problem I know that 00 is good, so I can declare L=0L=0 (it's possible to survive 0 days without food). I know that the biggest answer (max days) can be at most mm. But I don't know if mm works yet, so I need to assign R=m+1R=m+1.

Now if we go back to the problem F from yesterday's Educational round. I don't know yet if 00 or b[0]b[0] is good or bad, so I can't assign these values to LL or RR. So I need to assign −1−1 and b[0]+1b[0]+1.

The remaining thing you already determined correctly. I binary search over how much of b0b0 I assign to the city a0a0. Good (or possible) means, that the needs of all of the other cities a1,…an−1a1,…an−1 are satisfied. It's clear that assigning more to a0a0 will make it worse for the cities a1,…an−1a1,…an−1, the smaller assignments are good (possible), and the bigger ones are bad (not possible). Notice, that I accidentally used the wrong comments in my code. LL should be possible, and RR should be not possible.

I also cheated a little bit in the middle. Because even if the cities a1,…an−1a1,…an−1 are satisfied, the city a0a0 might not be. But it's easy to see, that if a solution exists, then it will be the one that assigns the most to a0a0 while still being possible for the other cities. So after the binary search LL is a solution, if a solution exists.
